Rip's heredocs (''' non-interpolating, """ interpolating) support visual margin control: when the closing delimiter is preceded only by whitespace, its column position becomes the "left margin" for all content lines. Text indented to the right of this baseline retains its relative indentation, while text at or left of the baseline is stripped to match it. This makes code generation elegant—just position the closing ''' or """ where you want the baseline, and the output aligns perfectly without manual indentation management.

• Two heredoc types: ''' (literal) and """ (interpolating)
• Visual margin control: Closing delimiter's column = left margin baseline
• Smart stripping: Only when closing delimiter has only whitespace before it
• Relative indentation preserved: Text right of baseline keeps its indent offset
• Perfect for codegen: Position closing delimiter → instant clean alignment
• Example: Closing at column 2 strips 2 spaces from all lines, closing at column 0 preserves all indentation

Actions in the grammar are simple.

1) empty value means use token 1 (1-indexed)
2) "" or null means use those as-is
3) a number 1 or 3 means use token 1 or 3
4) a string with numbers will be converted to the token position
5) if a bare number should be protected, then token numbers use a $n and bare numbers are untouched

--

Grammar actions in Rip use a simple, intuitive indexing system: empty/undefined defaults to token 1 (first matched symbol), literal "" or null pass through as-is, bare numbers (1, 3, etc.) reference token positions (1-indexed), and strings containing numbers automatically convert to token position references. For strings needing literal numbers, use $n syntax for token references (e.g., $1, $3) while bare numbers remain untouched—this prevents ambiguity between "return token 2" versus "return the number 2".

• Default (empty/undefined): Returns token 1 (first matched symbol)
• Literal values: "" or null pass through unchanged
• Bare numbers: 1, 3, etc. reference token positions (1-indexed)
• String actions: Numbers in strings auto-convert to token positions
• Protected literals: Use $n syntax ($1, $3) for tokens; allows bare numbers to stay literal

Rule examples with their action strings:

['Token1 Token2 Token3'] => action is Token1
['Token1 Token2 Token3', 'null'] => action is 'null' (pass literal value)
['Token1 Token2 Token3', 3] => action is Token3 (number -> token number)
['Token1 Token2 Token3', '2'] => action is Token2 (string -> token number)
['Token1 Token2 Token3', '[1, 3, 2]'] => action is [Token1, Token3, Token2] (array -> array of tokens)
['Token1 Token2 Token3', '[$3.slice(3,7), $2, 12]'] => action is [Token3.slice(3,7), Token2, 12] (same, protect bare numbers)

==[ Actions ]==

Default action (no action specified): Just returns token 1 (first symbol)

o 'IDENTIFIER' → returns the IDENTIFIER token itself (String object with all properties)
o 'Expression' → returns the Expression result


Actions reference token positions: 1, 2, 3 etc.

o 'Value . Property', '[".", 1, 3]' → builds array with tokens at positions 1 and 3
Position 2 (the . operator) is not referenced, so it's discarded


Protected references with $: $1, $2, $3 for literal positions

o 'Value INDEX_START RegexWithIndex INDEX_END', '[$3[0], $1, ...$3.slice(1)]'
The $ protects against auto-substitution


Tokens are passed through AS-IS: When an action references a token position, the entire token (with all its String object properties like .quote, .await, .heregex) is passed through.
